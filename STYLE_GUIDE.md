# Small Magellanic Cloud 设计宪章

Small Magellanic Cloud 的界面目标不是“炫”。它要像一个私人天文台：打开时有空间感，使用时足够安静，真正操作时又清楚、可靠、顺手。

## 北极星

> 让用户一眼知道自己在哪里，下一步能做什么，并且愿意多停留一会儿。

所有视觉、文案、动效、控件都服务这句话。任何元素如果只是在证明“我设计过”，就应该降级、合并或删除。

## 五个气质词

| 气质 | 解释 | 反面 |
| --- | --- | --- |
| 安静 | 不催促，不喧哗，不用持续高亮抢注意力 | 到处发光、浮动、解释自己 |
| 清楚 | 主体、状态、入口和反馈一眼可辨 | 低对比、透明过度、靠猜 |
| 深远 | 星云背景和空间层次建立个人宇宙感 | 背景压过正文 |
| 克制 | 色彩、阴影、圆角、动效只在必要处出现 | 每个元素都想当主角 |
| 有秩序 | 同类页面共享结构、命名、间距和交互 | 每页临时发明一套规则 |

## 判断顺序

1. 用户此刻要完成什么。
2. 当前屏幕的主信号是什么。
3. 哪些信息必须立刻可见。
4. 哪些信息可以弱化、压缩、延后或隐身。
5. 哪些元素只是重复、装饰或解释已明显的控件。
6. 深色、浅色、移动端是否仍然清楚。

不要先问“怎么更好看”。先问“什么该被看见”。好看来自正确的主次、比例和节奏。

## 页面类型

| 类型 | 密度 | 第一屏任务 |
| --- | --- | --- |
| 首页 | 低 | 建立身份与模块方向 |
| 模块入口页 | 中低 | 帮用户选择去哪里 |
| 学习工具页 | 中高 | 让用户直接下载、输入、运行 |
| 棋类游戏页 | 高 | 呈现棋盘、状态、模式和操作 |
| 设置弹层 | 紧凑 | 修改语言、外观等低频偏好 |

首页和模块页可以疏朗，工具和游戏必须有效率。不要把所有页面都做成同一种“漂亮卡片”。

## 同级页面一致性

同一个根路径下的同级页面必须像同一个系统里的房间，而不是临时拼接的页面。

- 同根同级页面共享页眉、面包屑、标题区、内容宽度、卡片半径、按钮语义和响应式断点。
- 功能可以不同，但相同位置的相似控件必须做相似的事：主按钮执行当前输入，示例按钮载入并运行示例，设置按钮只改偏好。
- 同级工具页的第一屏结构保持一致：标题说明在上，工作台在中，说明卡片在下；不要某一页突然把主操作藏进卡片底部。
- 同级入口页的卡片只负责导航；可点击卡片本身就是入口时，不再叠加重复按钮。
- 同级页面允许有不同命名，例如“转换”和“运行”，但它们在工具工作台里都必须是主执行按钮，位置、尺寸、禁用状态和加载反馈一致。
- 新增同级页面时，先复用现有组件类和语言键；只有出现真实的新交互语义，才新增样式或文案结构。

## 布局

- 外层留白建立气质，内部对齐建立效率。
- 同组元素靠近，不同组元素拉开。
- 宽屏优先增加阅读舒适度和并列信息，不盲目铺满。
- 移动端优先保证主体内容，不让页眉、弹层和低频控件挤压主要任务。
- 固定格式元素必须有稳定尺寸：棋盘、工具栏、设置项、卡片、计数器不能因文字变化跳动。

## 颜色与表面

- 背景是远景，内容表面是中景，文字和控件是近景；三层必须分清。
- 深色主题以星云、墨色、细线和少量青绿/琥珀强调为主。
- 浅色主题要干净、正式，不把深色主题简单反相。
- 透明材料必须有底。读不清时，提高实体感，不继续加模糊。
- 边框用于关系，阴影用于浮起，渐变用于气氛；不要用它们制造复杂感。
- 禁止让界面读成单一色系。青绿、蓝、琥珀是强调，不是铺满全站的染料。

## 排版

- 标题负责定位，不负责解释全部内容。
- 正文平实、准确、可信，不写营销式口号。
- 卡片摘要短，优先让用户扫描。
- 同类按钮使用同类动词，同类状态使用同类命名。
- 多语言保持同一人格：正式、友好、直接。
- 字号不要随视口宽度线性乱飘；容器内文字必须能放下，长词要能换行或压缩。

## 导航

- 页眉只放站点身份、必要导航和设置入口。
- 当前页已经由标题表达时，不重复堆面包屑当前项。
- 首页导航可以更轻，但不能让用户失去站点身份。
- 可点击卡片本身就是入口时，不再加“打开”“进入”等重复按钮。

## 设置面板

设置是低频但关键的基础设施。它必须稳定、干净、无残影。

- “语言”“外观”等左侧文字只说明设置项，不承担主点击目标。
- 右侧当前值是切换入口，点击后进入聚焦选择面板。
- 选项列表有返回入口、当前项标记和稳定高度。
- 图标使用干净单色图标，不使用 emoji，不使用会产生边缘纹理的 mask 叠加。
- 弹层要有足够实体感，不能和背景星云或棋盘混在一起。
- 桌面端和移动端使用同一套语义，移动端必须检查宽度、遮挡和误触。

## 学习工具

- 学习工具页比展示页更像工作台：少解释，多让用户操作。
- 下载卡片要明确平台、文件名、安装行为。
- 浏览器工具要翻译标题、标签、按钮、状态、错误和可访问标签。
- 示例按钮可以保留原始命令文本，因为命令本身就是内容。
- API 返回的底层错误可以保留技术信息，但常见空输入、过长、计算失败要本地化。

### 学习工具工作台

- 浏览器里可运行的学习工具统一使用工作台结构：输入区、示例区、主执行按钮、虚拟终端输出、补充说明。
- 主执行按钮使用同一视觉等级，放在示例按钮同一行的末端；移动端拉满宽度。
- 示例按钮的语义保持一致：点击后把示例写入输入区，并立即运行或转换，不能有的只填充、有的直接运行。
- 输入控件可以不同：单行命令用输入框，多行脚本用文本域；但标签、焦点态、间距和提交反馈一致。
- 虚拟终端必须保留真实终端排版：使用等宽字体、不启用自动折行、横向滚动承接长行，不把 SymPy/CLI 的字符画排版压成普通段落。
- 虚拟终端行高要接近真实 CLI，不能沿用正文的宽松行距；SymPy 分式、根式、级数等多行字符画需要上下行贴合。
- 虚拟终端宽度要尽量传给后端计算核心，例如通过 `COLUMNS` 控制 Python/SymPy 的 pretty print 列宽。
- 终端输出区域可以滚动，但不能因为 hover、运行状态或语言变化改变容器宽度。
- 工具页说明书使用逐层进入的链接结构：第一层只显示少量主题入口，进入下一层后再显示子主题或具体说明；不要把完整说明书一次性平铺在工具页，也不要用“核心来源、网页运行、本地命令”这类维护者说明卡代替用户说明书。说明书可见文案和可访问标签必须进入多语言字典。

## 棋类游戏

- 棋盘是主体，模式、状态、玩家、记录是辅助。
- 状态反馈要快而明确，不让玩家猜当前是否轮到自己。
- 联机房间信息要适合复制和口头传递。
- 导出、设置、帮助等低频操作收进轻量入口，不压过棋局。
- 棋子、墙、可走位、可吃子状态不能只靠颜色表达。

## 动效

- 动效短、轻、有方向，用来确认状态和空间关系。
- 背景微动可以存在，但必须慢、弱、可忽略。
- 弹层出现、二级选择、hover、focus 都要稳定，不制造布局跳动。
- 尊重 reduced motion；任何持续运动都必须有明确理由。

## 可访问性

- 文本对比度优先于漂亮背景。
- 颜色不能作为唯一状态表达。
- 所有可点击目标在移动端要足够大。
- 键盘焦点必须可见。
- 弹层不能困住焦点，也不能让焦点丢失。
- `aria-label`、`title`、placeholder 等也要进入语言体系。

## 信息处理策略

| 策略 | 用法 |
| --- | --- |
| 删除 | 无价值、重复、纯装饰 |
| 合并 | 意义接近的入口或状态 |
| 弱化 | 必要但低频的补充信息 |
| 压缩 | 有用但占空间的长说明 |
| 隐身 | 不影响当前任务的高级设置 |
| 提升 | 当前状态、关键错误、主要行动 |

如果一个元素无法归入这些策略，通常说明它还没有设计理由。

## 反模式

- 把高级理解成低对比、难读、虚。
- 把简洁理解成删掉必要入口、状态和反馈。
- 把正式理解成冷冰冰、无响应。
- 把美观理解成堆渐变、堆阴影、堆大图。
- 把统一理解成所有页面完全一样。
- 把隐身做成找不到。
- 把个人气质做成重复 logo、口号和装饰贴纸。

## 出手原则

改 UI 时，先做一轮减法，再做一轮对齐，最后才做视觉强化。
改语言时，先补字典，再补 DOM 标记，最后验证切换。
改文档时，先帮未来维护者定位，再记录细节。
